## 발생 오류

`.\start-dev.bat` 실행 시 아래와 같이 명령어 첫 글자가 잘려 오류 발생:

```
'AI'은(는) 내부 또는 외부 명령... 아닙니다.
'ot'은(는) 내부 또는 외부 명령... 아닙니다.
'opy'은(는) 내부 또는 외부 명령... 아닙니다.
'cho'은(는) 내부 또는 외부 명령... 아닙니다.
Error: [WinError 123] 파일 이름... 'backend\\1??'
```

## 원인 분석

### 1. UTF-8 한글 → CP949 충돌
이전 세션에서 Edit 도구로 배치 파일에 한글(`최초 1회`)을 추가했는데,  
Edit 도구는 UTF-8로 저장함.  
Windows CMD.EXE는 한국어 시스템 기본 인코딩인 **CP949(EUC-KR)**로 읽음.

→ 한글 UTF-8 바이트(`EC 88 9C ...`)를 CP949로 해석하면서  
   이후 ASCII 명령어까지 바이트 위치가 밀려 첫 글자가 잘리는 현상 발생.  
   (`not` → `ot`, `copy` → `opy`, `echo` → `cho`)

### 2. Unix 줄바꿈 (LF only)
파일 줄바꿈이 `LF(\n)`만 있었고, Windows CMD는 `CRLF(\r\n)`를 기대함.

---

## 해결 방법

### Step 1. 한글 제거
`echo   py -3.11 -m venv venv          (최초 1회)` 에서  
한글 `(최초 1회)` → 영문 `(first time only)` 로 교체

### Step 2. CP949 인코딩 + CRLF 줄바꿈으로 재저장

```powershell
$enc = [System.Text.Encoding]::GetEncoding(949)
[System.IO.File]::WriteAllText("...\start-dev.bat", $content, $enc)
```

- 인코딩: CP949 (Windows 한국어 ANSI, BOM 없음)
- 줄바꿈: CRLF (`\r\n`)
- 첫 바이트 확인: `0x40` (`@`) — BOM 없음 정상

## 교훈 / 주의사항

- **Windows 배치 파일(.bat)은 반드시 CP949 또는 ANSI 인코딩으로 저장**
- UTF-8 (BOM 유/무 관계없이) 한글 포함 시 CMD에서 오파싱 발생
- 줄바꿈은 CRLF 사용 (LF-only는 일부 CMD 버전에서 문제)
- Claude의 Edit/Write 도구는 UTF-8로 저장하므로,  
  한글이 포함된 배치 파일은 PowerShell의 `[System.IO.File]::WriteAllText(..., CP949 enc)`로 작성
