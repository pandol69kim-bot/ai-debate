## 진단 결과

### 버그 1 — React Strict Mode + hasConnected 가드 충돌 (결정적 원인) 🔴

**파일:** `frontend/src/app/debate/[id]/page.tsx`

**Next.js 15 기본 동작:** App Router에서 React Strict Mode가 **기본 활성화** (next.config.ts에 `reactStrictMode: false` 없음).

**React Strict Mode의 useEffect 실행 순서 (개발 모드):**

```
[1] Effect 실행: hasConnected.current = false → true, EventSource 생성
[2] Cleanup 실행: EventSource.close()  ← hasConnected.current는 여전히 true
[3] Effect 재실행: hasConnected.current === true → return;  ← EventSource 재생성 안 됨
```

**결과:** EventSource가 닫힌 뒤 재생성되지 않음 → `connected` SSE 이벤트 수신 불가 → store.status가 `"connecting"`에서 `"running"`으로 전환되지 않음 → 영원히 "연결 중..." 표시.

**수정 내용:**

```typescript
// 수정 전
const hasConnected = useRef(false);

useEffect(() => {
  if (!id || hasConnected.current) return;  // ← 두 번째 실행 때 여기서 막힘
  hasConnected.current = true;

  const es = createDebateStream(id);
  ...
}, [id]);

// 수정 후
useEffect(() => {
  if (!id) return;  // hasConnected 가드 제거

  const es = createDebateStream(id);
  esRef.current = es;

  es.onmessage = (e) => {
    try {
      const event = JSON.parse(e.data) as SSEEvent;
      useDebateStore.getState().handleSSEEvent(event);  // stale closure 방지
    } catch {}
  };

  es.onerror = () => { es.close(); };

  return () => {  // cleanup만으로 lifecycle 관리
    es.close();
    esRef.current = null;
  };
}, [id]);
```

**추가 개선:** `store.handleSSEEvent` → `useDebateStore.getState().handleSSEEvent` 변경으로 stale closure 문제도 방지.

### 버그 2 — `_debate_progress` 메모리 휘발 (구조적 문제) 🟡

**파일:** `backend/app/api/routes/debate.py`

```python
_debate_progress: dict[str, list] = {}  # ← uvicorn 재시작 시 초기화됨
```

**문제:** 토론 진행 이벤트가 서버 프로세스 메모리에만 저장됨. uvicorn 재시작, 서버 크래시 시 모든 진행 데이터 소실. 이미 완료된 토론의 SSE URL에 접속하면 `connected` 이벤트만 오고 이후 5분 대기 후 `timeout`.

**현재 영향:** 서버를 재시작하지 않는 한 진행 중인 토론에는 문제 없음. 완료된 토론의 SSE 재연결 시 문제 발생.

**이상적인 해결책 (미적용, 향후 개선):**
- Redis pub/sub로 이벤트 발행
- 또는 DB 폴링 방식으로 SSE 구현

**현재 임시 해결책:** 토론 완료 후에는 SSE 재접속 시도 대신 `GET /api/v1/debate/{id}` REST API로 기존 결과를 로드하는 로직 추가 권장.

### 버그 3 — timeout 이벤트 미처리 (소minor) 🟢

**파일:** `frontend/src/lib/store/debateStore.ts`

5분 대기 후 백엔드가 `{"type": "timeout"}` 이벤트를 전송하지만 프론트엔드 store switch문에 `case "timeout"`이 없어 무시됨.

**수정:** timeout 수신 시 "토론 응답 시간이 초과되었습니다." 에러 메시지와 함께 `status: "error"`로 전환.

## SSE 이벤트 흐름 정리

```
POST /debate/start → conversation_id 반환 → BackgroundTask 시작
  ↓
GET /debate/{id}/stream (SSE 연결)
  ↓ 즉시
{"type": "connected"} → status: "connecting" → "running"
  ↓ (배경 작업 진행에 따라)
{"type": "round_response", round_no, provider, content, ...} (여러 번)
{"type": "round_complete", round_no}
{"type": "judge_complete", winner, scores, summary}
{"type": "consensus_complete", final_answer, confidence_score}
{"type": "done"} → status: "done"
  또는
{"type": "error", message} → status: "error"
  또는 (5분 초과 시)
{"type": "timeout"} → status: "error" (수정 후)
```
